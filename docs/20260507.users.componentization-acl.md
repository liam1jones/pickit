## 1. Goal

Plan how to **extract user directories and role-aware behavior** from the monolithic app shell so that:

1. **Mock directories** (`USERS`, `CONTRACTORS`) live in small data modules, not in `App.jsx` / inline in `LoginScreen.jsx`.  
2. **Access control** is expressed in one place — derived from the authenticated session (including **portal** from the login flow) — instead of scattered string comparisons.  
3. **UI components** receive **capabilities** (or a thin `auth` / `session` object) rather than re-implementing rules from `user.role`.

This note ties the **ACL implied today** in `PickItApp` to the **session model** documented in the login-portals work.

---

## 2. Session model (from login portals)

After sign-in, `authedUser` passed into `PickItApp` as `initialUser` includes a **`portal`** discriminator:

| Portal | Shape (conceptual) |
|--------|---------------------|
| CoreWeave | `{ portal: "coreweave", name, role, email }` — `role` is one of the keys in `USERS` (`DCT`, `DCM / Tiger Team`, `ICS`). |
| Contractor | `{ portal: "contractor", name, role: "Contractor", contractorRole, company, email }` |

The root `App` wrapper keeps `key={authedUser.email}` so switching accounts resets state ([`20260505.auth.login-portals.md`](20260505.auth.login-portals.md) §2–3).

**Implication for componentization:** any “user module” should treat **portal + role (+ contractor fields)** as the stable **session** type; directory fixtures are only the source of valid emails for demo login.

## 3. Effective access control list today (`PickItApp`)

Rules are **not** centralized; they are inlined next to UI.

| Capability | Current rule (`App.jsx`) | Notes |
|------------|-------------------------|--------|
| **Approve ticket** | `user.role === "DCM / Tiger Team"` (`canApprove`) | Shown when `selT.status === "Pending Approval"`. |
| **ICS workflow** (pick/stage, close paths, excess confirm) | `user.role === "ICS"` (`canClose`) | Multiple buttons and copy (“Awaiting ICS…”) branch on `canClose`. |
| **Auto-approve on submit** | `user.role === "DCM / Tiger Team"` | Duplicated in `submitTicket` and form preview logic. |
| **Create project (`+ New`)** | `user.role === "ICS"` | Projects tab header. |
| **Assignee dropdowns in new-ticket form** | Lists from `USERS["ICS"]` and `USERS["DCM / Tiger Team"]` | Only meaningful for internal roles that file tickets. |
| **Demo “Switch role / user”** | `Object.entries(USERS)` | Builds menu of CoreWeave personas only; no contractor rows. |

**Contractors:** `LoginScreen` can sign in with `role: "Contractor"`, but the ACL above never grants `canApprove` or `canClose`. They effectively get a **view-heavy** path unless more checks are added later ([`20260505.auth.login-portals.md`](20260505.auth.login-portals.md) §5 — differentiate contractor UX).

**Presentation-only role maps:** `ROLE_COLOR` and `ROLE_DOT` in `App.jsx` key off the same three internal role strings; log lines use `user.role`. Contractors will fall through to defaults unless extended (e.g. amber styling by `portal === "contractor"`).

## 4. Componentization map (data → policy → UI)

### 4.1 Data layer (fixtures)

| Asset | Today | Target module (illustrative) |
|-------|--------|------------------------------|
| CoreWeave roster | `const USERS` in `App.jsx` (~line 152) | e.g. `src/data/mockCoreWeaveUsers.js` — same object shape, exported constant |
| Contractor roster | `CONTRACTORS` in `LoginScreen.jsx` | e.g. `src/data/mockContractors.js` |
| Portal chrome (optional) | `PORTALS` in `LoginScreen.jsx` | Keep colocated with login UI, or `src/data/loginPortals.js` if reused (e.g. marketing / help links) |

`LoginScreen` continues to accept `users={…}` from the app root, but the root imports the constant from `src/data/…` instead of closing over `USERS` inside `App.jsx`.

### 4.2 Access policy layer (new, small)

Introduce a **pure** module (no React), e.g. `src/auth/permissions.js` or `src/lib/access/pickitSession.js`, that exports functions from the **session** object:

- `canApproveTicket(session)`  
- `canPerformIcsActions(session)`  
- `canCreateProject(session)`  
- `shouldAutoApproveSubmission(session)`  
- Optionally: `isContractor(session)` → `session.portal === "contractor"` or `session.role === "Contractor"`

Implementation initially mirrors today’s strings; later, contractor-specific rules (filter tickets by `company`, hide destructive actions) extend this module only.

### 4.3 UI layer

| Surface | Refactor direction |
|---------|-------------------|
| `PickItApp` | Replace `const canApprove = …` / `canClose` with imports from the policy module; pass `session` (or `capabilities`) into large child components when extracted. |
| `Topbar` (user menu) | Receives `users` prop or imports fixture; for production, “switch user” may disappear — keep behind dev/demo flag if needed. |
| `FormModal` / new ticket | Receives `icsOptions` and `dcmOptions` derived from `USERS` keys, or receives pre-built arrays from a `getAssigneeOptions()` helper next to fixtures. |
| `LoginScreen` | Import `CONTRACTORS` from data module; unchanged props contract `users`, `theme`, `onLogin`. |

---

## 5. Dependency order (recommended)

1. **Extract fixtures** — `USERS`, `CONTRACTORS` (and optionally `PORTALS`) so `App.jsx` and `LoginScreen.jsx` only import. Zero behavior change.  
2. **Add policy functions** — swap `canApprove` / `canClose` / auto-approve / “+ New” to use those functions; still zero UX change for CoreWeave roles.  
3. **Align contractor styling** — extend `ROLE_COLOR` / log labels via policy or session helpers so activity feed and header do not show empty styling for contractors.  
4. **Split UI components** — as in the separation-of-concerns note: `Topbar`, modals, and detail actions receive `capabilities` or destructured booleans from the parent that computed them once from `session`.

---

## 6. Contracts to preserve

- **Role keys** in the CoreWeave directory object must stay exactly `DCT`, `DCM / Tiger Team`, and `ICS` until ticket logic and seed logs are migrated to a richer enum or IDs.  
- **`onLogin` payload** must remain compatible with [`20260505.auth.login-portals.md`](20260505.auth.login-portals.md) §3 (portal, contractor fields).  
- **Form assignee `<select>`** uses `value={u.name}` today; moving `USERS` must not change `name` strings without updating seed tickets / `INIT_*` references.

---

## 7. Follow-ups (from login portals + this split)

1. **Contractor-specific experience** — implement in policy + views (filtering, badges), not in fixture files alone.  
2. **Real IdP** — replace `attempt()` in `LoginScreen` and swap fixture imports for API/context; keep **policy module** consuming the same session shape.  
3. **Persist session** — localStorage rehydrate ([`20260505.auth.login-portals.md`](20260505.auth.login-portals.md) §5.2); validate portal/role on load before trusting stored JSON.
