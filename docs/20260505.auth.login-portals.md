## 1. Goal

Add authentication to PickIT and split the entry experience into two distinct portals so internal CoreWeave staff and external contractors each get a tailored sign-in flow.

---

## 2. Phase 1 — Initial login screen

### What was built
A login screen now gates the entire app. On first load the user sees a sign-in card; after authenticating they land in the existing dashboard.

### New file: `src/pages/LoginScreen.jsx`
A self-contained, dark-themed login card matching the rest of the UI:
- **PickIT** wordmark + "powered by CoreWeave" lockup at the top.
- Email + password fields with inline validation (empty fields, unknown email).
- Primary **Sign in** button that validates the entered email against the existing `USERS` directory (any password is accepted in this demo build — no real auth backend yet).
- Collapsible **Quick demo sign-in** list showing every demo user (DCT, DCM / Tiger Team, ICS) so a reviewer can jump in as any role with one click.

### Wiring in `src/App.jsx`
- The previous `App` component was renamed to `PickItApp` and now accepts `initialUser` and `onLogout` props.
- A new default-exported `App` is a thin wrapper that owns the auth state and either renders `LoginScreen` or the full app:

```jsx
export default function App(){
  const [authedUser,setAuthedUser]=useState(null);
  if(!authedUser)return <LoginScreen users={USERS} theme={D} onLogin={setAuthedUser}/>;
  return <PickItApp key={authedUser.email} initialUser={authedUser} onLogout={()=>setAuthedUser(null)}/>;
}
```

- `key={authedUser.email}` ensures all in-app state resets cleanly when the user signs out and back in as someone else.

### Sign out
The existing user-menu dropdown in the topbar gained a **Sign out** row (red accent) that invokes the new `onLogout` prop and returns the user to the login screen.

## 3. Phase 2 — Two-portal experience (CoreWeave + Contractor)

The login flow became a two-step experience.

### Step 1 — Portal chooser (new first screen)
A centered card with two side-by-side tiles:
- **CoreWeave** — blue accent (`#1E40F6`), sub-label *"Internal team — DCT, DCM / Tiger Team, ICS"*.
- **Contractor** — amber accent (`#f59e0b`), sub-label *"Vendor & partner field teams"*.

Each tile has its own icon badge, hovers to its accent color, and clicking it advances to the matching sign-in form. Footer hint: *"Need access? Contact your CoreWeave program manager."*

### Step 2 — Per-portal sign-in form
Both forms share the same chrome (PickIT logo, email, password, error banner, primary CTA, demo quick-pick) but adapt to the chosen portal:
- Header shows the portal's icon badge + label, e.g. **"CoreWeave portal"** or **"Contractor portal"**.
- The submit button picks up the portal's accent color.
- A `← Choose a different portal` link lets the user step back without reloading.
- The email placeholder switches between `you@coreweave.com` and `you@vendor.com`.
- Validation/lookup runs against the appropriate directory:
  - CoreWeave → existing `USERS` map.
  - Contractor → new `CONTRACTORS` list defined at the top of `LoginScreen.jsx`.

### Shape of the user object passed to `onLogin`
Now carries the portal context so downstream code can branch on it:

```jsx
// CoreWeave sign-in
{ portal: "coreweave", name, role, email }

// Contractor sign-in
{ portal: "contractor", name, role: "Contractor", contractorRole, company, email }
```

`App.jsx` accepts this object verbatim — no further wiring needed — and `key={authedUser.email}` on `PickItApp` continues to reset state cleanly on logout.

## 5. Open / suggested next steps

1. **Differentiate the contractor experience inside the app.** Both portals currently drop into the same dashboard. Possibilities:
   - Hide approval/close buttons (already gated by role; contractors with `role: "Contractor"` should naturally see view-only).
   - Show a `"Contractor — {company}"` badge in the topbar instead of the role pill.
   - Filter tickets to only those tied to the contractor's `company`.
2. **Persist auth across reloads** by writing `authedUser` to `localStorage` and rehydrating it from `useState`'s initializer.
3. **Wire a real auth provider** — Okta / Azure AD for CoreWeave, magic-link or vendor SSO for contractors — by replacing the body of `attempt()` in `LoginScreen.jsx`.
4. **Optional: remember last-used portal** so returning users skip the chooser.
