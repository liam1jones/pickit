# PickIT — Location-Scoped Access Control List (ACL) Design

**Date:** 2026-05-05
**Section:** Auth
**Subsection:** Access Control
**Status:** Design / implementation guide (not yet wired into the app)

**Related docs:**
- `docs/20260505.auth.login-portals.md` — current login + portal flow this design extends.

**Source files referenced:**
- `src/lib/cores/locations/index.ts` (`ALL_LOCATIONS`, `LOCODES`)
- `src/lib/cores/locations/types.ts` (`CoreWeaveLocation`, `SiteLogistics`)

---

## 1. Goal

Today every authenticated user can see every ticket and every site in PickIT. We need to add a **location-aware ACL** so each user only sees and acts on data tied to the sites they own. This document specifies:

1. The roles we are introducing and how each one maps to a permission profile.
2. The data model changes needed on the user object and on `CoreWeaveLocation`.
3. Where in the codebase to enforce reads (filtering) and writes (authorization).
4. A migration path that ships incrementally without breaking the demo build.

---

## 2. Role catalog

| Role key (in code)       | Display name                  | Portal      | Default location scope                               |
| ------------------------ | ----------------------------- | ----------- | ---------------------------------------------------- |
| `accountant`             | Accountant                    | CoreWeave   | Read-only across **all** sites (financial reporting) |
| `ics`                    | Inventory Control Specialist  | CoreWeave   | Sites where the user is listed as `ics` or `regionalIcm` |
| `dcm`                    | Data Center Manager           | CoreWeave   | Sites where the user is the regional manager (`regionalIcm`) plus any site explicitly assigned |
| `dct`                    | Data Center Tech              | CoreWeave   | A single home site (and any sites explicitly assigned) |
| `vendor`                 | Vendor / Contractor           | Contractor  | Only sites where the vendor has an active engagement |

> The existing `USERS` map in `src/App.jsx` (line 152) already groups members under `DCT`, `DCM / Tiger Team`, and `ICS`. The new schema replaces those free-form labels with the canonical keys above so authorization is deterministic.

## 3. Permission matrix

Permissions are expressed as `{ read, create, approve, pickStage, close, exportFinancials, manageVendors }` flags. *S* = scoped to allowed sites only, *G* = global (all sites), *—* = denied.

| Capability                               | Accountant | ICS | DCM | DCT | Vendor |
| ---------------------------------------- | ---------- | --- | --- | --- | ------ |
| View tickets / projects / work orders    | G (read)   | S   | S   | S   | S (only their own) |
| Create ticket                            | —          | S   | S   | S   | S      |
| Approve ticket                           | —          | —   | S   | —   | —      |
| Pick / Stage materials                   | —          | S   | —   | —   | —      |
| Receive materials & start work           | —          | —   | —   | S   | S (own work only) |
| Close ticket                             | —          | S   | S   | —   | —      |
| Export financial reports / NetSuite sync | G          | —   | S   | —   | —      |
| Manage vendor roster                     | —          | —   | G   | —   | —      |
| Edit site metadata (`CoreWeaveLocation`) | —          | —   | S   | —   | —      |

The matrix is the single source of truth and lives in code as a constant — see §6.

## 4. Data model changes

### 4.1 Extend `CoreWeaveLocation` with an ACL block

`src/lib/cores/locations/types.ts` already carries `regionalIcm` and `ics` strings. Add an optional `acl` block so each site file can declare its own access list without changing the main shape:

```ts
export type LocationAcl = {
  /** Emails of DCMs allowed full management of this site. */
  dcms: string[];
  /** Emails of ICS users (and RICMs) allowed pick/stage/close. */
  ics: string[];
  /** Emails of DCTs whose home site is this location. */
  dcts: string[];
  /** Vendor emails or vendor-company domains (e.g. "lumaoptics.com"). */
  vendors: string[];
};

export type CoreWeaveLocation = {
  // ...existing fields...
  acl?: LocationAcl;
};
```

Each site file (e.g. `src/lib/cores/locations/US-CVG01.ts`) gains an `acl` block:

```ts
export const location: CoreWeaveLocation = {
  code: "US-CVG01",
  label: "US-CVG01 – Cincinnati, OH",
  timezone: "America/New_York",
  zipCode: "45202",
  regionalIcm: "Liam Jones",
  ics: "Alissah Wiles",
  address: "401 Kingsview Dr, Lebanon, Ohio 45036",
  acl: {
    dcms: ["ljones@coreweave.com"],
    ics:  ["awiles@coreweave.com"],
    dcts: ["jtorres@coreweave.com"],
    vendors: ["lumaoptics.com", "spatel@lumaoptics.com"],
  },
};
```

> Vendors can be either an exact email or a company domain. The resolver in §5 normalizes both.

### 4.2 Replace free-form `USERS` with a typed roster

Today `USERS` is a flat dictionary keyed by display label. Replace it with a typed array sourced from a single file (e.g. `src/lib/auth/users.ts`) so the resolver can join against it:

```ts
export type RoleKey = "accountant" | "ics" | "dcm" | "dct" | "vendor";

export type AppUser = {
  email: string;
  name: string;
  role: RoleKey;
  /** For vendors only. */
  company?: string;
  /** Optional explicit overrides — sites the user can access regardless of ACL files. */
  siteOverrides?: string[]; // LOCODE list, e.g. ["US-CVG01"]
};

export const ROSTER: AppUser[] = [
  { email: "ljones@coreweave.com", name: "Liam Jones",   role: "dcm" },
  { email: "awiles@coreweave.com", name: "Alissah Wiles", role: "ics" },
  { email: "jtorres@coreweave.com", name: "J. Torres",   role: "dct" },
  { email: "spatel@lumaoptics.com", name: "Sam Patel",   role: "vendor", company: "Luma Optics" },
  { email: "finance@coreweave.com", name: "Accounting",  role: "accountant" },
];
```

The `LoginScreen` already produces a payload with `email`, `role`, and (for contractors) `company`. The login step becomes: look the email up in `ROSTER`, attach the canonical `RoleKey`, and hand the resolved `AppUser` to the rest of the app.

## 5. Resolver — turning a user into a site list

Create `src/lib/auth/acl.ts` as the **only** module that knows how to compute "what can this user see?". Everything else asks this module instead of recomputing scope inline.

```ts
import { ALL_LOCATIONS, type CoreWeaveLocation } from "../cores/locations";
import type { AppUser, RoleKey } from "./users";

const ROLE_SCOPE_FIELD: Record<RoleKey, keyof NonNullable<CoreWeaveLocation["acl"]> | "all" | "none"> = {
  accountant: "all",   // global read
  dcm:        "dcms",
  ics:        "ics",
  dct:        "dcts",
  vendor:     "vendors",
};

export function allowedSitesFor(user: AppUser): CoreWeaveLocation[] {
  const field = ROLE_SCOPE_FIELD[user.role];
  if (field === "all")  return ALL_LOCATIONS;
  if (field === "none") return [];

  const overrides = new Set(user.siteOverrides ?? []);
  return ALL_LOCATIONS.filter((loc) => {
    if (overrides.has(loc.code)) return true;
    const acl = loc.acl;
    if (!acl) return false;
    const list = acl[field];
    if (!list) return false;
    if (user.role === "vendor") {
      const domain = user.email.split("@")[1]?.toLowerCase();
      return list.some((entry) => {
        const e = entry.toLowerCase();
        return e === user.email.toLowerCase() || e === domain;
      });
    }
    return list.map((e) => e.toLowerCase()).includes(user.email.toLowerCase());
  });
}

export function canSeeSite(user: AppUser, locode: string): boolean {
  return allowedSitesFor(user).some((l) => l.code === locode);
}

export function canDo(
  user: AppUser,
  action: "create" | "approve" | "pickStage" | "close" | "exportFinancials" | "manageVendors",
  ctx?: { locode?: string },
): boolean {
  // Site scope check first — everything except global read needs a site to be valid.
  if (ctx?.locode && !canSeeSite(user, ctx.locode)) return false;

  switch (action) {
    case "create":            return user.role !== "accountant";
    case "approve":           return user.role === "dcm";
    case "pickStage":         return user.role === "ics";
    case "close":             return user.role === "ics" || user.role === "dcm";
    case "exportFinancials":  return user.role === "accountant" || user.role === "dcm";
    case "manageVendors":     return user.role === "dcm";
  }
}
```

Two functions, one matrix — that is the entire authorization surface.

## 6. Where to enforce in the existing UI

### 6.1 At login — resolve and attach
In `src/pages/LoginScreen.jsx`, replace the inline lookup (against `USERS` / `CONTRACTORS`) with a single call into the new roster + resolver:

```jsx
import { ROSTER } from "../lib/auth/users";
import { allowedSitesFor } from "../lib/auth/acl";

const match = ROSTER.find((u) => u.email.toLowerCase() === target);
if (!match) { setErr("No PickIT account found for that email."); return; }
const sites = allowedSitesFor(match);
if (sites.length === 0 && match.role !== "accountant") {
  setErr("Your account has no sites assigned. Contact your DCM.");
  return;
}
onLogin({ ...match, allowedLocodes: sites.map((s) => s.code) });
```

The `App` wrapper now stores `authedUser.allowedLocodes` and threads it through context.

### 6.2 Filtering reads
Wrap the existing ticket/project arrays with a memoized scope filter at the top of `PickItApp` in `src/App.jsx`:

```jsx
const visibleLocodes = useMemo(
  () => new Set(user.allowedLocodes),
  [user.allowedLocodes],
);
const visibleTickets = useMemo(
  () => tickets.filter((t) => visibleLocodes.has(t.site.split(" ")[0])),
  [tickets, visibleLocodes],
);
const visibleProjects = useMemo(
  () => projects.filter((p) => visibleLocodes.has((p.site || "").split(" ")[0])),
  [projects, visibleLocodes],
);
```

Then swap every existing reference to `tickets`/`projects` in the rendered JSX (board view, table view, analytics tabs) for the `visible*` versions.

### 6.3 Filtering site selects
The `<select>` dropdowns at lines 525, 1226, and 1426 in `src/App.jsx` currently render every entry from `LOCODES`. Replace them with the scoped list:

```jsx
const visibleLocodeLabels = useMemo(
  () => LOCODES.filter((label) => visibleLocodes.has(label.split(" ")[0])),
  [visibleLocodes],
);
// ...
{visibleLocodeLabels.map((l) => <option key={l} value={l}>{l}</option>)}
```

### 6.4 Authorizing writes
Every state mutator that today checks role inline (`approveTicket`, `closeTicket`, `pickStage`, etc.) gets a one-line guard:

```jsx
if (!canDo(user, "approve", { locode: selT.site.split(" ")[0] })) return;
```

Buttons that trigger those actions should also be hidden via the same `canDo` check so the UI stays consistent.

### 6.5 Hiding navigation
- **Vendors:** hide the analytics tab and limit the board to the "My work" filter.
- **Accountants:** hide ticket creation; expose a new "Financials" view that calls `canDo(user, "exportFinancials")`.
- **DCT:** hide the project-management view; show only the board scoped to their home site.

## 7. Vendor portal specifics

The contractor login already produces `{ portal: "contractor", company, ... }` (see Phase 2 in `20260505.auth.login-portals.md`). To make ACLs work for vendors with minimum churn:

1. The roster maps every contractor email to `role: "vendor"` and stores `company`.
2. Each `LocationAcl.vendors` array can list either specific emails or a company domain.
3. In `allowedSitesFor`, the vendor branch matches the email **or** its domain — this lets a DCM grant access to "everyone @ Luma Optics" with one entry.
4. Vendors only see tickets where they are the requestor (`t.req` matches their name) **and** the site is in their allowed set. Add a second filter to `visibleTickets` for vendors:

```jsx
const scopedForVendor = user.role === "vendor"
  ? visibleTickets.filter((t) => t.req === user.name)
  : visibleTickets;
```

## 8. Audit logging

Every existing ticket already carries a `log: [{ ts, who, role, action, detail }]` array (see `T0`–`T9` in `src/App.jsx` lines 158–167). Extend it with two fields:

```ts
type LogEntry = {
  ts: string;
  who: string;
  role: RoleKey;
  action: string;
  detail: string;
  locode?: string;     // resolved site, for cross-site reporting
  allowed: boolean;    // result of the canDo() check at the time of write
};
```

When a `canDo` check returns `false`, still write a log row with `allowed: false`. That gives security/operations a tamper-evident trail of denied attempts without breaking the user flow.

## 9. Migration plan

Ship in four PR-sized steps so the demo build never breaks:

1. **PR 1 — types + roster (no behavior change).**
   Add `LocationAcl` to `types.ts`, create `src/lib/auth/users.ts`, mirror the existing `USERS` content into `ROSTER`. Login still uses the old map.
2. **PR 2 — resolver + login wiring.**
   Add `src/lib/auth/acl.ts`, change `LoginScreen` to look up via `ROSTER`, attach `allowedLocodes` to the user object. The app still ignores it.
3. **PR 3 — read-side filtering.**
   Add `visibleTickets` / `visibleProjects` / scoped site selects. UI now hides out-of-scope data.
4. **PR 4 — write-side guards + UI gating.**
   Insert `canDo` checks in every mutator and hide buttons accordingly. Roll out vendor-only filtering and the new accountant view.

Each PR can be merged independently and reverted without rolling back the others.

## 11. Open questions

1. Should `accountant` be allowed to drill into a single-site view, or always remain at portfolio level?
2. Do we want **per-site role** (e.g. someone is a DCT at `US-CVG01` but a DCM at `US-CMH01`)? The current model assumes one role per user globally; if not, expand `AppUser` to `roles: { locode: RoleKey }[]` and update `allowedSitesFor`.
3. Where does the source of truth live long-term — in-repo TS files, NetSuite, or an external IdP group claim? The resolver above is the abstraction boundary; pick whichever backend without changing call sites.
4. Should denied actions surface a toast (`"You don't have permission for this site"`) or stay silent? Current proposal: silent UI hide + audit-log row.
