## 1. Background

The current PickIT prototype (`src/App.jsx`) ships as a self-contained React app that simulates the full materials-pick lifecycle for CoreWeave data centers — a Data Center Tech (DCT) submits a request for parts, the DCM / Tiger Team approves it, the Inventory Control Specialist (ICS) picks and stages the materials, the DCT receives them and starts the work, and any unused stock is returned. The prototype already encodes:

- A canonical 7-state status machine: `Open / Submitted → Pending Approval → Approved – Pending Transfer → Picked / Staged → In Progress – Work Underway → Excess Return Pending → Resolved / Closed` (`src/App.jsx:19`).
- A directory of 50+ sites with regional ICM, ICS, address, and logistics metadata under `src/lib/cores/locations/`.
- Three internal personas (DCT, DCM / Tiger Team, ICS) and a contractor portal scaffold (`src/pages/LoginScreen.jsx`).
- Hooks for downstream systems (NetSuite bin transfers, Asana task links).

The prototype proves the workflow but is not production-grade: it has no persistent backend, no real auth (see `20260505.auth.login-portals.md`), no live inventory feed, and no scoped access (see `20260505.auth.access-control.md`). To get to a usable MVP **without** building all of that ourselves, we will implement the same workflow on top of two tools CoreWeave already runs — **Jira** for ticketing and approvals, and **Tableau** for inventory visibility — and use PickIT only as the thin "DCT cockpit" UI that wraps both.

## 2. Problem statement

DCTs currently rely on a mix of email, Slack threads, and Asana tasks to request parts. There is no single source of truth for the state of an inventory request, no consistent SLA on approvals, and no live view of on-hand stock that a tech can check before walking out to a rack. ICS staff spend significant time chasing missing context ("which rack? which DH?"), and DCMs cannot tell at a glance which sites are over- or under-stocked. The prototype proved the workflow shape; the MVP must turn it into a production system that integrates with the tools CoreWeave already trusts.

## 3. Goals & non-goals

### 3.1 Goals (MVP scope)
- **G1.** Every materials request lives as a Jira issue in a dedicated **PickIT** project, with a workflow that mirrors the prototype's 7 states.
- **G2.** DCTs submit requests through a streamlined PickIT form (or, as a fallback, directly in Jira) and get notified on every state change.
- **G3.** ICMs / ICSs approve, pick/stage, and close tickets from Jira — no parallel system.
- **G4.** A Tableau dashboard, embedded inside PickIT, gives DCTs live inventory levels for the parts catalog at their assigned site(s) before they submit a ticket.
- **G5.** Site, role, and vendor scoping match the ACL design in `20260505.auth.access-control.md`: a DCT only sees their site's tickets and inventory; vendors only see their own work.
- **G6.** End-to-end median time-to-approval drops below **30 minutes** during business hours.

### 3.2 Non-goals (deferred past MVP)
- Native mobile app (responsive web only).
- Automated procurement / re-order triggers (Tableau will surface low stock; ordering still manual).
- Replacing NetSuite as the system of record for inventory.
- Vendor self-service ticket creation (read-only vendor view in MVP).
- Offline mode for warehouse floors.

## 5. User stories

1. As a **DCT** at `US-CVG01`, I open PickIT and see the live SFP-10G-SR on-hand count for my DH before opening a ticket — so I don't request something that's already on the shelf.
2. As a **DCT**, I submit a request with site, DH/rack, part, qty, and an Asana link in **under 30 seconds**, and I get a Jira issue ID back.
3. As a **DCM**, my Jira queue shows only `Pending Approval` tickets for my region, sorted by SLA risk, and I can approve with one click.
4. As an **ICS**, when a ticket flips to `Approved – Pending Transfer`, I get a Slack DM and see it appear on my site's pick board within seconds.
5. As an **ICS**, when I scan the QR tag on the staged kit, the ticket auto-advances to `Picked / Staged`.
6. As a **DCT**, when I confirm "Materials received," NetSuite fires a bin transfer (`Stored → In Process`) and the ticket moves to `In Progress – Work Underway`.
7. As an **ICS**, I can flag excess return, capture used vs. returned counts, and the ticket closes after the return is confirmed.
8. As an **Accountant**, I can pull a Tableau export of every closed ticket in a date range with the linked NetSuite transaction IDs.
9. As a **DCM**, the Tableau dashboard shows me cycle time per state, broken down by site, for the last 30 days.
10. As a **vendor**, I sign in to the contractor portal and see only the tickets where I'm the requestor at the sites my company is cleared for.

## 7. Jira implementation

### 7.1 Project & issue type
- New Jira project: **PickIT** (key: `PICK`). Issue ID format `PICK-####` matches the prototype's existing format (`T0`–`T9` in `src/App.jsx`).
- Single issue type: **Materials Request**.

### 7.2 Custom fields
Mapped 1:1 from the React state shape so we can move data both directions without translation:

| Jira custom field | Type | Source in prototype |
| ----------------- | ---- | ------------------- |
| `cw_site` | Single-select (LOCODE list, populated nightly from `src/lib/cores/locations/index.ts`) | `t.site` |
| `cw_dh_rack` | Short text | `t.location` |
| `cw_part_lines` | JSON / multi-row | `t.lines[]` |
| `cw_qty_requested` | Number | `t.qtyReq` |
| `cw_qty_installed` | Number | `t.qtyIns` |
| `cw_qty_returned` | Number | `t.qtyRet` |
| `cw_ics_owner` | User picker (filtered to ICS group) | `t.ics` |
| `cw_dcm_owner` | User picker (filtered to DCM group) | `t.dcm` |
| `cw_qr_payload` | Short text (QR contents) | `t.qrPayload` |
| `cw_asana_link` | URL | `t.asana` |
| `cw_netsuite_txn` | Short text (NetSuite transaction ID once fired) | derived |

### 7.3 Workflow (states + transitions)

The Jira workflow is the prototype's 7 states verbatim (`src/App.jsx:19`):

| Jira status | Transition | Allowed actor (post-MVP ACL) | Trigger |
| ----------- | ---------- | --------------------------- | ------- |
| Open / Submitted → Pending Approval | `Submit` | DCT | Auto on issue create |
| Pending Approval → Approved – Pending Transfer | `Approve` | DCM / Tiger Team | Manual (button in Jira) |
| Pending Approval → Open / Submitted | `Send back` | DCM | Manual, requires comment |
| Approved – Pending Transfer → Picked / Staged | `Pick & stage` | ICS | Manual, or auto on QR scan |
| Picked / Staged → In Progress – Work Underway | `Materials received` | DCT | Manual; fires NetSuite `Stored → In Process` |
| In Progress – Work Underway → Excess Return Pending | `Flag excess` | ICS / DCT | Manual, requires used vs. returned counts |
| In Progress – Work Underway → Resolved / Closed | `Close (no excess)` | ICS | Manual |
| Excess Return Pending → Resolved / Closed | `Confirm return` | ICS | Manual; fires NetSuite `In Process → Stored` |

DCM / Tiger Team leads keep the existing **auto-approve-on-self-submit** shortcut from the prototype (`src/App.jsx:388`) via a workflow post-function that triggers `Approve` immediately when the reporter is a member of the DCM group.

### 7.4 Boards
- **DCM queue** — JQL `project = PICK AND status = "Pending Approval" AND cw_site IN (regionalIcmSites(currentUser()))`.
- **ICS pick board** (per region) — JQL `status IN ("Approved – Pending Transfer", "Picked / Staged") AND cw_ics_owner = currentUser()`.
- **DCT my-work** — JQL `reporter = currentUser() AND statusCategory != Done`.

### 7.5 SLAs
- DCM approve: **30 min** business hours / 4 hr off-hours.
- ICS pick: **2 hr** after approval.
- Total cycle (submit → resolved): **24 hr** target, **48 hr** breach.

## 8. Tableau implementation

### 8.1 Data source
A nightly + on-event sync from `pickit-svc` lands the following tables in the Tableau data source `pickit_inventory`:

- `inventory_on_hand` — `{ site_locode, part_sku, dh, on_hand_qty, reserved_qty, available_qty, last_synced_at }` (sourced from NetSuite, refreshed every 15 min).
- `pickit_tickets` — flattened Jira issue history (`{ ticket_id, site, part, qty_req, qty_ins, qty_ret, status, opened_at, approved_at, picked_at, closed_at, dct, dcm, ics }`).
- `site_directory` — projection of `ALL_LOCATIONS` (`{ locode, label, regional_icm, ics, timezone, country }`).

### 8.2 Dashboards
1. **DCT — "What's on the shelf?"** (default landing view, embedded in PickIT)
   - Inputs: site (auto-set from logged-in user's home site), part search.
   - Tiles: on-hand qty per part by DH, 30-day burn rate, "next likely outage" forecast based on slope.
   - Drill: click a part → recent tickets that consumed it.
2. **ICS — "Pick load today"**
   - Open tickets at site, age in current state, parts that are below safety stock.
3. **DCM — "Regional ops"**
   - Cycle-time funnel (median time per state), SLA breaches, top requesters, excess-return rate per ICS.
4. **Accounting — "Inventory movement"**
   - Closed tickets joined to NetSuite transactions, $ value moved per site per month, open Excess Return Pending balance.

### 8.3 Embedding inside PickIT
- Tableau workbooks are embedded via the **Tableau Embedding API v3** in a new `<TableauPanel/>` React component placed on the PickIT board view.
- `pickit-svc` mints a short-lived **Tableau Connected App JWT** that asserts the user's email and the role keys defined in `20260505.auth.access-control.md`. Tableau row-level filters use those claims to restrict each user to their allowed `site_locode` set.

## 9. Integration service (`pickit-svc`)

A new service (initially Node + Fastify, deployed alongside the existing app) owns:

- **Jira webhook receiver** — listens for issue create / transition / comment, normalizes to PickIT events, fans out to NetSuite + Slack + Tableau extract refresh.
- **NetSuite client** — fires bin transfers on `Materials received` and `Confirm return`, stamps `cw_netsuite_txn` back on the Jira issue.
- **Slack notifier** — DMs the assigned ICS on approval, the DCT on pick, and the DCM channel on SLA breach.
- **Tableau JWT minter** — for embedded views.
- **ACL resolver** — implements `allowedSitesFor(user)` and `canDo(...)` from `20260505.auth.access-control.md`. PickIT's UI calls `pickit-svc` instead of computing scope client-side.

## 10. Authentication & authorization

- **SSO:** CoreWeave staff via Okta (or Azure AD) — same provider that gates Jira and Tableau, so a single sign-in covers all three surfaces.
- **Vendors:** keep the contractor portal from `20260505.auth.login-portals.md`; back it with magic-link email auth in `pickit-svc` for MVP.
- **Authorization:** roles + per-site scope from `20260505.auth.access-control.md`. The Jira `cw_site` field is the join key; the resolver answers "which LOCODEs is this user cleared for?" and both Jira JQL filters and Tableau row-level filters consume that list.

## 11. Data flow — happy path

1. **DCT** signs in to PickIT, lands on the Tableau "What's on the shelf?" view filtered to their site.
2. They confirm SFP-10G-SR is below their threshold and open the request form.
3. PickIT POSTs to `pickit-svc /tickets`, which creates a Jira issue in `Open / Submitted` and immediately transitions it to `Pending Approval`. Issue ID returned to the UI.
4. Jira webhook → `pickit-svc` → Slack DM to the on-call DCM with one-click approve link.
5. **DCM** approves in Jira → status `Approved – Pending Transfer` → Slack DM to the assigned ICS.
6. **ICS** picks/stages → scans the QR tag → ticket flips to `Picked / Staged` (QR scan endpoint hits `pickit-svc /pick`).
7. **DCT** confirms received in PickIT → `pickit-svc` transitions Jira to `In Progress – Work Underway` and posts the NetSuite bin transfer `Stored → In Process`.
8. **DCT** completes work; **ICS** flags excess (or closes clean). NetSuite `In Process → Stored` posts on excess return. Ticket closes.
9. Tableau extract refreshes within 5 min; "What's on the shelf?" reflects the new on-hand count.

## 13. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
| ---- | ---------- | ------ | ---------- |
| Jira workflow customization gets pushback from Atlassian admins | Medium | High | Engage Jira admins in design review week 1; use a project-scoped workflow scheme so it doesn't touch global config. |
| Tableau extract latency makes "live" inventory view stale | High | Medium | 15-min on-hand refresh + on-event refresh after each `In Progress` transition; UI shows "as of HH:MM". |
| NetSuite write failures leave Jira/inventory out of sync | Medium | High | All NetSuite calls go through `pickit-svc` with idempotency keys; failures stay queued and ticket banner shows "NetSuite sync pending". |
| ACL drift between Jira groups, Tableau data-source filters, and `pickit-svc` resolver | High | Medium | Single source of truth = the location ACL files (`src/lib/cores/locations/*.ts`); a nightly job pushes group membership into Jira/Tableau from those files. |
| DCT adoption — "I'll just keep emailing" | Medium | High | Sunset email path 4 weeks post-launch; auto-reply emails with a link to the PickIT form. |
| Vendor data exposure across companies | Low | High | Row-level Tableau filter on company domain + audit log of every vendor view (per access-control doc §8). |

## 15. Open questions

1. **Inventory grain** — Tableau "on-hand" should it be at the LOCODE level or DH level? DH-level matches the prototype's `t.location` granularity but doubles the data volume. Proposal: DH-level for the DCT view, LOCODE rollup for the DCM view.
2. **Auto-approve threshold** — should DCM auto-approve apply to small qty (< 5 of any part) regardless of requester? Today the prototype only auto-approves when the requester is a DCM lead.
3. **Embedded Jira vs. linked Jira** — embed the Jira issue panel inside PickIT or just link out? Embedding is faster for DCTs but adds Atlassian licensing cost.
4. **Vendor write access** — MVP says read-only, but Phase 1 feedback may justify letting vendors mark "On site, ready to install" via a single transition. Defer the decision to Phase 2 review.
5. **International sites (`GB-*`, `ES-*`, `NO-*`, `SE-*`, `DK-*`, `CA-GAL01`)** — SLA targets and Tableau timezone handling need a separate review before Phase 3.
6. **Source of truth for parts catalog** — today `src/lib/catalog` ships statically. Should the MVP read from NetSuite item master nightly? Recommended yes.

### 16.3 Glossary

- **DCT** — Data Center Tech. Submits requests.
- **DCM** — Data Center Manager / Tiger Team. Approves requests, owns SLA.
- **ICS** — Inventory Control Specialist. Picks, stages, closes tickets.
- **RICM** — Regional ICS Manager (a senior ICS).
- **LOCODE** — site code, e.g. `US-CVG01`. Defined per file under `src/lib/cores/locations/`.
- **DH** — data hall.
- **Bin transfer** — NetSuite inventory movement between logical bins (`Stored`, `In Process`).
